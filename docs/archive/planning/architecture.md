# ModPack IDE 架构设计

> 专为 Minecraft 整合包开发设计的 AI IDE，类似 Cursor/Trae 但深度集成游戏数据。

---

## 1. 概述

### 1.1 定位

整合包开发者的运行时工作台。不是替代 IntelliJ/Gradle，而是提供：
- 实时游戏数据连接
- AI 辅助的配方/物品/标签管理
- 内置知识图谱支撑的语义理解
- 材质可视化优先的交互体验

### 1.2 核心原则

1. **材质是第一语言** — 视觉识别优于 ID 辨别，所有 UI 组件优先显示物品图标
2. **被动层始终在线** — Agent 无任务时，IDE 本身提供自动补全、校验、提示
3. **渐进式复杂度** — 从轻量连接开始，按需扩展能力

### 1.3 技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| 桌面壳 | Electron | 跨平台桌面应用 |
| UI 框架 | React + TypeScript | 组件化开发 |
| 编辑器 | Monaco Editor | 代码编辑核心，自定义 Language ID |
| 构建工具 | pnpm + Turborepo | Monorepo 管理 |
| 持久化 | SQLite | 图数据 + 向量 + 元数据 |
| 向量检索 | sqlite-vss | 零依赖嵌入式向量搜索 |
| 游戏连接 | WebSocket | 实时双向通信 |

## 3. 数据层

### 3.1 Exporter Mod

**现有能力（复用）：**
- 数据采集：ModListSource、ItemRegistrySource、RecipeSource、ItemTagSource、ItemResourceSource
- 客户端渲染：ItemRenderHelper（OpenGL FBO → Base64 PNG）
- 持久化：SQLite 导出

**需要改造：**
- 添加 WebSocket Server
- 实现 hash-based diff 检测
- 材质按需渲染（替代全量导出）

### 3.2 WebSocket 协议

**同步流程：**
```
游戏 reload
  ↓ Exporter 重算所有 hash (< 1s)
  ↓ 推送 diff 给 IDE
  ↓ IDE 对比，找出变化项
  ↓ 请求变化项完整数据
  ↓ 增量更新图谱 + 向量索引
```

**消息类型：**
- `sync_request` — IDE 请求全量/增量同步
- `diff_response` — Exporter 返回 hash diff
- `data_response` — Exporter 返回完整数据
- `texture_request` — IDE 请求按需渲染
- `texture_response` — Exporter 返回渲染结果

## 4. 知识层

### 4.1 Graph Engine (知识图谱)

**节点类型：**
- Item（物品）
- Recipe（配方）
- Tag（标签）
- Mod（模组）

**边类型：**
- `CRAFTED_FROM` — 配方输入→物品
- `USED_IN` — 物品→配方引用
- `TAGGED_UNDER` — 物品→标签归属
- `PROVIDED_BY` — 物品→所属 mod

**存储：**
- SQLite 表：`nodes(id, type, properties_json)` + `edges(source_id, target_id, type, properties_json)`
- 内存图：启动时加载，热查询走内存
- 增量同步：WebSocket 推送时更新

### 4.2 Vector Search (语义检索)

**数据源：**
- Mod 文档/Wiki 切片
- 配方描述文本
- 物品 tooltip
- 社区知识（常见整合套路）

**用途：**
- 自然语言查询 → 相关物品/配方
- 模糊匹配（"自动化矿物处理" → 多个 mod 方案）
- AI 上下文注入

**技术：**
- sqlite-vss：零依赖，与图数据库统一技术栈
- 本地嵌入模型（ONNX Runtime + GPU）

### 4.3 Texture Service (材质服务)

**接口：** `getItemTexture(itemId) → Promise<Image>`

**特性：**
- 懒加载：按需渲染当前视口内的物品
- 缓存层：内存缓存 + 磁盘持久化
- Hash 去重：相同贴图只存一份
- 批量预取：预测用户可能查看的物品

## 6. Agent 层

### 6.1 三层架构：Task → Skill → Tool

```
Task（任务） — 用户目标
  ↓ 需要什么能力？
Skill（技能） — 可复用的领域知识
  ↓ 调用什么具体功能？
Tool（工具） — 具体函数
```

**示例：**
```
Task: "统一所有铜锭"
  ↓
Skill: ItemUnification
  ├── 分析 tag 分组
  ├── 对比材质
  ├── 推荐首选项
  └── 生成配置
  ↓
Tools:
  ├── search_items("copper")
  ├── get_recipes("forge:ingots/copper")
  ├── compare_items([...])
  └── generate_kubejs_script({...})
```

### 6.2 设计原则

**Skill 设计：**
- 原子化 — 一个 Skill 只做一件事
- 可组合 — 多个 Skill 可以串联完成复杂任务
- 幂等性 — 相同输入产生相同输出
- 无状态 — 不依赖上次执行结果

**Tool 设计：**
- 单一职责 — 一个 Tool 只做一个操作
- 可验证 — 返回结构化结果，可被上层检查
- 零副作用 — 只读操作不修改状态
- 批量支持 — 支持单个和批量模式

### 6.3 两层体验

| 层级 | 触发方式 | 能力 | Token 消耗 |
|------|---------|------|-----------|
| 被动层 | 始终在线 | 自动补全、校验、材质预览、跨引用提示 | 零 |
| 主动层 | 用户请求 | 配方分析、批量操作、冲突检测、脚本生成 | 按需 |

### 6.4 Tool 集

专为整合包设计的内置工具：

| 工具 | 功能 | 类型 |
|------|------|------|
| `search_items` | 按名称/标签/mod 搜索物品 | 只读 |
| `get_recipes` | 获取物品的所有配方 | 只读 |
| `analyze_recipe_tree` | 分析配方依赖链 | 只读 |
| `detect_conflicts` | 检测配方冲突 | 只读 |
| `compare_items` | 对比多个物品的配方差异 | 只读 |
| `get_texture` | 获取物品材质预览 | 只读 |
| `generate_unified_recipe` | 生成统一配方 | 生成 |
| `generate_kubejs_script` | 生成 KubeJS/CraftTweaker 脚本 | 生成 |
| `generate_gatekeeper_config` | 生成物品移除/清理配置 | 生成 |
| `generate_resource_override` | 生成资源覆盖文件 | 生成 |

### 6.5 Skill 集

可复用的操作封装：

| Skill | 功能 | 复杂度 |
|-------|------|--------|
| **物品合并** | 同名物品统一为一个 | 中 |
| **配方迁移** | 将 A 物品的配方迁移到 B | 中 |
| **冲突解决** | 自动检测并建议修复配方冲突 | 高 |
| **批量修改** | "所有 Create 配方能源消耗降低 20%" | 中 |
| **文档生成** | 自动生成整合包 Wiki/说明 | 低 |
| **物品变换** | 隐藏/换皮/重塑物品 | 高 |

### 6.6 Token 优化

**大部分功能不需要 Agent：**
- 自动补全 → 规则匹配
- Schema 校验 → 纯逻辑判断
- 材质预览 → 渲染服务
- 跨引用提示 → 图查询

**知识层减少 token 消耗：**
- 语义搜索（本地，零 token）
- 图查询（本地，零 token）
- 只把精准上下文注入 AI（几百 token）

## 7. 性能设计

### 7.1 增量同步

整合包规模可达万级物品 × 50 配方 = 50 万配方节点。全量重建不可接受，必须增量同步。

**Exporter 端 diff 检测：**
- Hash-based：每次 reload 重算所有数据的 hash，与 IDE 存储的 hash 对比
- 只推送 hash 变化的部分
- Hash 计算（SHA256）50 万配方 < 0.5 秒（内存计算）

**SQLite 写入优化：**
- 单事务批量插入（10 万条 < 1 秒）
- WAL 模式（并发读写）
- 索引延迟创建（先插入完再建索引）

**全量首次同步优化：**
- 分批推送（每批 1000 条，边推边加载）
- gzip 压缩（压缩率 70-80%）
- 多线程并行查询 Registry

### 7.2 GPU 加速

| 操作 | 适合 GPU | 说明 |
|------|---------|------|
| 向量嵌入 | ✅ 加速 10-20x | 批量并行计算，GPU 优势明显 |
| 向量相似度搜索 | ✅ 加速 5-10x | 批量距离计算 |
| Hash 计算 | ❌ CPU 够快 | 无需加速 |
| 图遍历 | ❌ 内存瓶颈 | 随机访问模式，GPU 无优势 |
| 批量图查询 | ✅ 可选 | 多物品并行遍历时有收益 |

**嵌入模型方案：ONNX Runtime + GPU**

```
ONNX Runtime (Node.js binding)
├── NVIDIA GPU → CUDA
├── Apple Silicon → CoreML/Metal
├── AMD GPU → DirectML
└── 无 GPU → CPU fallback
```

### 7.3 性能目标

| 操作 | 目标时间 |
|------|---------|
| 首次全量构建（1万物品） | < 30 秒 |
| 增量更新（100个配方变更） | < 1 秒 |
| 物品搜索（模糊匹配） | < 50ms |
| 配方树遍历（3层深度） | < 100ms |
| 向量相似度查询 | < 100ms |
| 单物品嵌入（GPU） | < 10ms |

## 8. 与现有方案对比

### 8.1 Almost Unified

**核心机制：**
- Tag 分组（`forge:ingots/copper` 等 common tag）
- 每组指定首选项
- 运行时拦截配方注册，替换为首选物品
- 过滤 JEI/REI 显示

**我们采纳：**
- Tag 分组逻辑 → 知识图谱的 Tag 关系建模
- 首选项概念 → 用户选择保留哪个
- 配方替换逻辑 → Agent 生成替换脚本

**我们改进：**
- 可视化对比材质（而非盲选）
- AI 推荐统一方案（而非手动配置）
- 离线预览影响（而非运行时才看到效果）

### 8.2 OEI

**核心机制：**
- 隐藏列表
- JEI/REI 显示过滤
- 轻量级，不改配方

**我们采纳：**
- 轻量模式 → 简单场景用 OEI 式配置，不触发 Agent
- 显示过滤 → IDE 物品列表同样支持隐藏/显示

**我们改进：**
- 统一管理隐藏 + 配方（OEI 只管显示）
- 批量操作（OEI 只能单个配置）

### 8.3 整合策略

```
简单场景（只隐藏显示）
  → 生成 OEI 式配置
  → 零 Agent 消耗

中等场景（统一配方）
  → 分析 tag 分组
  → 用户选择首选项
  → 生成 Almost Unified 配置 或 KubeJS 脚本

复杂场景（深度魔改）
  → AI 分析影响范围
  → 推荐方案
  → 生成完整资源覆盖
```

## 9. 使用场景

### 9.1 场景 1：统一同名物品

**问题：** 多个 mod 提供名称相似但玩法不同的物品（如"精炼铁锭"）

**流程：**
1. 语义搜索 → 找出所有相似物品
2. 并排显示材质 → 用户选择保留哪个
3. 知识图谱 → 展示每个物品的配方树
4. AI 建议 → "保留 A，迁移 B 和 C 的配方"
5. 用户确认 → 生成统一配方 JSON
6. 一键推送 → 游戏 reload 生效
