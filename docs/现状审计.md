## 一句话结论

这是一个**「只读浏览器」已成型、「智能编译器」尚未动工**的早期应用。

- 真正能跑通的闭环：*导入附属 Mod 预导出的 `export.sqlite` → 浏览物品 / 配方 / 项目*。
- 规格（`docs/spec-snapshot/`）里定义 ModPack IDE 的一切核心 —— Agent 语义识别、`unify` 引擎、置信×风险门、决策清单、diff、KubeJS 产物 —— **代码里一行都没有**。
- 仓库还背着大量 Delightify 旧时代的**死代码**（JAR 字节码分析器、JAR 解析批量写入、孤立的 LLM 服务）。

---

## 关键发现（与文档 / 直觉冲突的真相）

1. **导入的不是 JAR，是预导出的 SQLite。**
   `services/mod-data-importer/importer.ts` 读取整合包里 `delightify-exporter/export.sqlite`（附属 Mod 在游戏内导出），把 `mods / items / item_tags / recipes / item_resources` 原样搬进 `project.db`。**不解析 JAR、不解析字节码。**

2. **Java 字节码分析器是死代码。**
   `packages/main/java-tools/` 和 `packages/main/java/mod-inspector/`（两套，后者是更旧的残留）在整个 TS 端**零调用**（无 `spawn` / `execFile` / `.jar` 引用）。属于 Delightify 旧路线遗物。

3. **`services/database/batch-save.ts` 整个文件是死代码，且 schema 对不上。**
   它往 `items` 插 `display_name / category / texture_path...`、还引用 `translations / textures` 表 —— 这些列和表在当前 schema 里**根本不存在**。全仓无人 `import` 它。旧 JAR 解析时代遗留。

4. **LLM 服务是孤儿。**
   `services/llm/`（OpenAI / Anthropic / Ollama 三 provider 真能调）**没有任何 IPC / 页面调用它**，且它干的活是分析「JAR 注册模式」(`registrationPattern: deferred_register / static_field...`)，属旧路线，**不是规格里的 Agent**。

5. **两份 schema 定义，且工作区表没被建。**
   Drizzle `schema.ts` 定义了 `recipe_edits / export_history`，但实际建表走的是 `schema-manager.ts` 的 `CORE_TABLES`，那里**没有这两张表**。所以"配方编辑历史""导出历史"在运行时**根本不会被创建**。

6. **没有任何 KubeJS / 产物生成。**
   全仓搜 `kubejs / datapack / emitter` 只命中：schema 的一个 `enum`、i18n 文案、和两个**纯占位**页面（RecipeEditor / ConversionTool）的按钮（点了没反应）。

## 对 MVP-0（设计/10）的影响

MVP-0 = *离线灌一个真实整合包 → 目录 / 图谱 → 端到端跑通"统一同名物品" → 可审 diff → KubeJS 产物*。

  1. `unify` 引擎（同名物品聚类 → 生成动作）

- **建议清理**：删 `batch-save.ts`、两套 Java 工程；或明确把孤立的 `llm/` 标记为"待重写为 Agent"，以免后续误以为已实现。

## 数据来源：Delightify-Exporter（附属 mod）

本地 repo：`/Users/aeroseira/SSD-1/WorkDir/IdeaProjects/Delightify-Exporter`（Forge 1.20.1 / Java 17，GPL-3.0，远端 `github.com/Aero-Seira/Delightify-Exporter`）。
游戏内 `/delightify_export dump` → 写出 `<world>/delightify-exporter/export.sqlite`（schema v4），即 IDE importer 的唯一数据来源。

**Exporter 导出的表（权威 schema，`db/Schema.java`）：**

| 表 | 内容 | 备注 |
|---|---|---|
| `manifest` | exported_at_utc / minecraft_version / forge_version / world_name | KV |
| `mods` / `items` / `item_tags` | 注册表最终态 | — |
| `recipes` | recipe_id / type_id / modid / hash / **raw_json** / unparsed | 仅原始 JSON |
| `item_resources` | lang_name、texture(Base64)、texture_path、texture_main、model、blockstate… | texture/model 内容仅客户端环境可得 |
| `recipe_views` | per-type 布局 JSON + 渲染 PNG | M4，仅客户端 |
| `recipe_view_backgrounds` | 背景 PNG BLOB | 仅客户端 |

**对 MVP-0 至关重要的两点：**

1. **配方无结构化输入输出。** Exporter 文档把 `recipe_inputs/recipe_outputs` 列为 "v1+"，但 `Schema.java` 未建这两张表 —— 配方只有 `raw_json`（含 `ingredients`/`result`/`pattern`）。**unify 引擎必须自行解析 `raw_json`**，两端都无结构化层。

2. **导入端与导出端不对齐（IDE importer 有遗漏）：**
   - `importer.ts:717` 把 `item_resources` 过滤成只留 `texture` + `lang_name`，丢弃 `texture_main`/`blockstate`/`model`/`texture_path` —— 方块"正面材质"图标策略数据被扔掉。
   - importer 完全不读 `recipe_views` / `recipe_view_backgrounds` —— 配方可视化渲染数据进不了 project.db。
