# Delightify 项目计划（开发路线 / Roadmap）

> 本文档用于指导 Delightify 的阶段性开发与迭代节奏。目标是让我们每次推进都有明确的"范围、验收标准、交付物与风险控制"。  
> 当前日期：2026-03-15

---

## 0. 项目目标与边界

### 0.1 产品目标（一句话）
为整合包开发者提供一个"本地化、可视化、可审计"的 Minecraft 魔改工作台：导入模组 JAR → 建立知识库 → 可视化浏览/编辑配方 → 导出为 KubeJS / Datapack（LLM 仅作为可选辅助，不替代人工裁决）。

### 0.2 关键设计原则（必须长期坚持）
1. **本地优先**：默认不依赖云端，数据与实例在本地闭环。
2. **三层路径分离（PCL2CE 风格）**：应用数据（userData）/ 项目数据（.delightify）/ 整合包文件（kubejs/mods/config）。
3. **global 推荐 + project 覆盖**：跨项目可复用的知识放 `global.db`，项目裁决放 `project.db`。
4. **可解释与可审计**：任何自动建议（规则或 LLM）都要能追溯来源、版本与理由。
5. **可扩展的配方类型元数据**：`config/recipe_types` 可编辑、可扩展，优先做到"新增 JSON 就能支持新机器/新类型"。

### 0.3 非目标（至少 v0.x 不做）
- 纯云端 SaaS 模式（可作为未来选项）
- 运行时反射读取 Forge/Fabric 注册表（不引入 JVM）
- 一开始就支持所有魔改类型（附魔/战利品表/进度等先放后面）

## 3. M1：全局知识库与 JAR 导入闭环（v0.2）✅ 已完成

### 3.1 目标
用户能够导入一个模组 JAR，并在 `global.db` 里看到结构化数据（哪怕覆盖率不完美）。

### 3.2 已完成范围
- ✅ `global.db` 初始化与迁移（Drizzle）
- ✅ JAR 解压读取（zip-reader）
- ✅ 三策略提取：lang / tags / recipes
- ✅ textures 落盘缓存 + DB 元信息
- ✅ 导入进度通过 IPC 推送到 UI（进度条）
- ✅ 内置配方类型初始化
- ✅ 模组删除功能（级联删除关联数据）

### 3.3 关键决策（已拍板）
- ✅ **textures 存储策略**：落盘缓存 + DB 元信息
  - 实际 PNG 文件存储在 `userData/cache/textures/`
  - 数据库只存储元信息（cache_name, file_hash, width, height）
  - 优势：加快迭代、减少 DB 膨胀、便于调试

### 3.4 验收标准 ✅
- ✅ 导入任意一个常见模组 JAR（例如 Farmers Delight）不会崩溃
- ✅ `mods/items/recipes/translations/tags/textures` 数据写入并持久化
- ✅ UI 能看到"导入中/完成/失败"的状态与错误信息
- ✅ 支持增量导入（重复导入同一模组会更新数据）

### 3.5 技术实现要点
- **数据库**：libsql + Drizzle ORM，支持所有核心表的 CRUD
- **JAR 解析**：纯 Node.js 实现，无需 JVM
  - Lang 解析：从 `assets/{modid}/lang/*.json` 提取物品名称
  - Tag 解析：从 `data/{modid}/tags/items/*.json` 提取标签关联
  - Recipe 解析：支持原版及模组自定义配方类型
  - Texture 提取：使用 jimp 处理 PNG 文件
- **IPC 通信**：完整的进度回调机制，支持取消操作
- **数据完整性**：外键约束 + 级联删除

## 6. M4：可视化配方编辑 + 导出（v0.5）

### 6.1 目标
实现"从 UI 编辑 → 写回配方文件 → 导出脚本"的闭环，哪怕只支持少量配方类型。

### 6.2 范围
- RecipeEditor：槽位点击选物品、数量、tag/nbt（先 minimal）
- Export：导出 KubeJS（优先）与 Datapack（可后置）
- 项目目录写入：写到 `<modpack_root>/kubejs/`（遵守路径分层）
- Snapshots：导出快照与回滚机制（最简版：时间戳目录）

### 6.3 验收标准
- 编辑一条配方后能导出成可用脚本（用户手动放进实例可运行）
- 同一配方多次导出不会覆盖丢失（可通过 snapshots 解决）

## 7. M5：语义关系系统（overlay）+ 转换辅助（v0.6）

### 7.1 目标
把"推荐"与"裁决"分离：global 提供候选，project 保存用户最终选择，形成可复用的知识沉淀。

### 7.2 范围
- `global.db` 与 `project.db` 的语义关系表（或至少一张通用表）
- 合并规则：project > global；project 可显式屏蔽 global
- UI：在需要选择/映射的场景展示候选与置信度，并允许"确认/拒绝/编辑"

### 7.3 验收标准
- 对同一关系：global 中存在推荐，project 确认后以后默认用 project
- 切换项目不会污染别的项目裁决，但能复用 global 推荐

## 8. M6：LLM 批量迁移工作流（可选）（v0.7）

### 8.1 目标
LLM 帮你"省时间"，但你永远有最终控制权（审阅工作流）。

### 8.2 范围
- llm:convert：批量输入 → 流式进度 → 产出候选建议
- 置信度阈值：低于阈值必须人工审阅
- 所有输出都通过 M5 的语义关系系统落到 project（形成裁决）

### 8.3 验收标准
- 同一批任务可暂停/取消/重试
- LLM 失败不影响本地核心功能（可完全关闭 LLM）

## 10. 迭代节奏与任务拆分方式（我们怎么"逐步推进"）

### 10.1 每轮迭代建议
- 迭代周期：3~7 天（按你的节奏）
- 每次只做一个"可验收闭环"
- 每次迭代产出：
  1) 变更列表（PR）
  2) 验收清单（勾选）
  3) 风险/未决项

### 10.2 下一步（建议从这里开始）
建议我们把 **M2** 拆成 3 个可立即开干的小步：
1. Items 查询 IPC 完善与测试
2. ItemIcon 组件开发（材质读取 + 降级显示）
3. ItemBrowser 页面实现（搜索、筛选、分页）

## 11. 风险清单（提前标红）

- **纹理存储策略**：存 DB 会让 global.db 迅速膨胀；落盘缓存要处理失效与清理。
- **跨模组配方引用**：recipe 扫描会出现别的 modId 的 item；需要允许 items 表存在"外部引用占位"或延迟补全。
- **配方类型爆炸**：早期不要追求覆盖所有格式；优先做元数据驱动与可扩展机制。
- **LLM 不确定性**：必须把 LLM 产出当"候选"，所有落库必须可追溯、可撤销。
