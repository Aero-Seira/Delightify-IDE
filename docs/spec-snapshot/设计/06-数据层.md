# 数据层 —— v0 草案

> **状态**：v0 草案。回答"系统需要哪些关于整合包的数据、各从哪来、怎么建库"。**离线优先 hybrid**。
> ⚠️ 现实基线（已核对源码仓库）：现有 JAR 分析器（`packages/main/java-tools/.../com.delightify.modinspector`）**只提取物品/方块注册 + 多方块**，**不含配方/tag**。本层的核心新增工作就是补上配方/tag/显示名的提取。

---

## 1. 系统需要哪些数据

| 数据 | 用途 |
|---|---|
| **物品/方块目录** | id、显示名、所属 mod、所属 tag、是否可放置 | 浏览、选择器、爆炸半径 |
| **配方** | 输入/输出、类型/机器、数量 | 一切动作的操作对象 |
| **tag 及成员** | tag→物品 映射 | 选择器、unify/retag、输入集校正 |
| **显示名(lang)** | 各语言名称 | 语义辨识、rename、UI |
| **材质/模型** | 图标 | "材质是第一语言"、UI |
| **图谱关系** | 配方↔物品↔tag↔mod 的边 | 选择器 by_graph、爆炸半径、链分析 |

---

## 2. 来源映射（关键）

| 数据 | 来源 | 现状 |
|---|---|---|
| 物品/方块注册 | **字节码分析**（DeferredRegister 等） | ✅ 现有分析器已做 |
| 多方块结构 | 字节码启发式 | ✅ 现有 |
| **配方** | JAR 内 `data/<ns>/recipes/*.json` + datapack | ❌ **需新增 JSON 提取** |
| **tag 成员** | JAR 内 `data/<ns>/tags/**/*.json` + datapack | ❌ **需新增 JSON 提取** |
| **显示名** | JAR 内 `assets/<ns>/lang/*.json` | ❌ **需新增（简单）** |
| 材质/模型 | JAR 内 `assets/<ns>/textures|models/` | 文件可取；渲染需客户端 → §5 |
| **缺口**（任何来源都拿不到） | KubeJS 脚本生成的配方/tag、运行时 tag 合并、datapack 加载顺序覆盖的**最终结果** | → §6 可选 runtime 通道 |

> 好消息：配方/tag/lang 都是 **JAR 里的纯 JSON 文件**，离线解压即可读，比字节码分析简单得多。

---

## 3. 离线优先 hybrid 管线

```
整合包目录（mods/*.jar + datapacks + 可选 kubejs/）
  │
  ▼ ① 字节码分析（复用现有 Java CLI：Main.java → JSON）   → 物品/方块注册
  ▼ ② JSON 提取（新增，Node 侧解压 JAR/datapack）          → 配方 / tag / lang
  ▼ ③ 合并 + 解析引用（item id ↔ tag ↔ 配方）              → 知识图谱 / 目录
  ▼ ④ 写入 libsql（@libsql/client）                        → 可查询的本地库
  ▼ ⑤ 材质：批量渲染或资源包提取 → 磁盘缓存（§5）
  │
  └─（可选，非 v1 关键路径）runtime 通道补 §2 缺口 → 增量校正
```

**②是本层主要新增**：Node 侧读 JAR（zip）里的 `data/<ns>/recipes`、`data/<ns>/tags`、`assets/<ns>/lang`，比扩展 Java 分析器更轻。

---

## 4. 知识图谱 / 目录 schema（建于 libsql）

- **节点**：`Item/Block`、`Recipe`、`Tag`、`Mod`。
- **边**：`CRAFTED_FROM`（配方→输出物）、`USED_IN`（物品→引用它的配方）、`TAGGED_UNDER`（物品→tag）、`PROVIDED_BY`（物品→mod）；以及链分析用的 `produces/consumes/depends_on`。
- **存储**：`nodes(id, type, props_json)` + `edges(src, dst, type, props_json)`，热查询走内存。
- **服务选择器**（[[03-Intent-Spec契约|设计/03 §3]]）：by_id/tag/mod/type/graph 直接落到图查询；by_name_semantic 走 lang 文本匹配（+ 可选语义）；by_semantic_family/equivalence_class/category_template 由 LLM 在图上成组（[[09-LLM-Agent集成]]）。
- v1 可能用 **SQLite FTS5 + tag/名称匹配** 即够，向量检索推迟（见 [[doc-set-status]] 技术纠正）。

---

## 5. 材质策略

- 渲染依赖**活动游戏客户端**（最重依赖）→ 不走"实时渲染"，改 **一次性批量渲染 + 磁盘缓存**（hash 去重）。
- 退路：多数物品/方块材质可从 `assets/.../textures` + 模型 JSON **离线合成**（无需启动游戏），复杂模型再回退到批量渲染。

---

## 6. 静态 vs 运行时边界（诚实划清）

- **离线能拿**：JAR/datapack 里声明的物品、配方、tag、lang、材质——覆盖绝大多数整合包内容。
- **离线拿不到**：KubeJS/CraftTweaker 脚本在运行时生成的配方/tag、Forge/Fabric 的 tag 运行时合并结果、多 datapack 加载顺序覆盖后的**最终态**。
- **策略**：v1 **离线为主**；上述缺口由**可选 runtime 通道**（游戏内导出，非 v1 关键路径）增量校正。若某整合包重度依赖 KubeJS 动态生成，再启用 runtime。

---

## 7. 待打磨

- KubeJS 脚本的**静态解析**可行性（部分 `event.recipes.*` 调用可静态读出）——能省掉一部分 runtime 依赖。
- "是否可在世界放置"（爆炸半径用，[[04-引擎与动作语义|设计/04 §4]]）：从方块注册即可保守判定。
- 增量同步（整合包改动后只更新 diff）——v1 可先全量重建，规模允许时再优化。
- ②的 JSON 提取放 Node 侧 vs 扩展 Java 工具：倾向 Node 侧（轻、与主进程同栈）。
