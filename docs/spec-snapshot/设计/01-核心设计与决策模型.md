## 1. 一句话定位

作者只给 **目标**（如"优化整合包内所有面食的配方"），Agent **自主感知整合包内容、对每条配方逐案做语义判断、规划如何在全包忠实落实**；高置信决策自动执行（仍逐条可审），不确定的**自动搁置成待裁决队列**，在任务其余部分完成后交作者定夺。

> 目的：**省去作者完全掌握整个整合包生态的时间与脑力开销。**

它不是会设计的 AI，也不是花哨的批处理器——而是一个**理解你意图、并替你把意图在整个包里落实**的执行型 Agent。

---

## 2. 关键分界（本项目的立身之本）

| 能力 | 例子中的体现 | 归谁 | 可行性 |
|---|---|---|---|
| **(a) 语义辨识** | 知道面粉≠面团≠生面条≠发酵面团；知道"魔法转换"不是烹饪 | **Agent** | ✅ 可行、可验证 |
| **(b) 全包执行规划** | 自己找出所有相关配方、逐条判断改不改/换成谁、连带清理冗余 | **Agent** | ✅ 可行 |
| **(c) 设计 / 平衡判断** | "面食值不值得优化"、整合包是写实还是魔法风、成本砍多少 | **作者** | Agent 不碰 |

GPT-5.5 评审否定"语义理解"，错在把 (a)(b) 和 (c) 捆在一起。**拆开后，(a)(b) 是 LLM 的强项且能做安全，(c) 本就是作者的——矛盾消失。**

> **用户拥有「目标」，Agent 拥有「如何在一个你没空全记住的整合包里忠实落实这个目标」。**

## 3. 核心管线

```
目标（自然语言，可不完整）
  │
  ▼ 分流   过宽/几乎全是设计判断(c) 的请求（如"前期太肝，降一档"）
  │        → 转「引导式规划」：只整合并呈现相关信息、引导作者规划，Agent 不自行规划执行
  │        其余 → 走下面的执行管线 ↓
  │
  ▼ 阶段0  目标解释：Agent 写下它对目标的理解，作为头号"可改假设"（不阻塞）
  │
  ▼ 阶段A  范围召回：把目标展开成种子概念 → 经名称/tag/lang/图谱邻域捞出全部候选
  │         （优化"召回"，宁可多捞；并亮明搜索范围，可补漏）
  │
  ▼ 阶段B  逐案判断：对每个候选 →【先分类】这是什么 →【再定动作】→【给置信×风险】
  │         高置信 & 低风险      → 自动决策（仍逐条可审）
  │         不确定 / 高风险/高爆炸半径 → 自动搁置
  │
  ▼ 决策清单 + 待裁决队列   ★闸门1：人扫一遍，纠错 + 集中处理搁置项
  ▼ 引擎执行 → 完整 diff     ★闸门2：确认效果
  ▼ 产物（KubeJS / datapack / Almost Unified …，多后端）
```

两道闸门 = 完全可控：闸门1 在零计算成本时抓"理解错"，闸门2 抓"结果错"。

## 4. 决策模型

- **先分类，再动作**：Agent 不直接"把这条改了"，而是先产出 `{对象=煮面步骤, 置信=高}`，再据分类选动作。**分类是人能一眼复核的最小单元**，是语义判断"可审计而非黑箱"的关键。
- **路由 = 置信 × 风险双门**（不只看置信）：

  | | 低风险（如替换原料，可逆） | 高风险（如删物品、改 tag，跨 mod 影响） |
  |---|---|---|
  | **高置信** | ✅ 自动（可审） | ⚠️ 搁置（附引用清单/爆炸半径） |
  | **中/低置信** | ⚠️ 搁置（附选项+推荐） | ⚠️ 搁置 |

- **待裁决队列**：每个搁置项 = `{情形, 选项A/B/C, Agent 暂定推荐, 为何不确定}`。任务其余先完成，集中交作者。
- **置信不靠 LLM 自报**：用**多信号一致性**——名称语义 + tag + 配方类型/机器 + 图谱位置，四者同向→高置信，相互冲突→自动降级搁置。可解释、更可靠。
- **召回透明**：阶段A 必须亮明"经哪些种子/tag 命中 N 条"，杜绝静默截断导致的"不完整优化"。
- **级联与爆炸半径**：动作有连带后果（替换→孤立物品→可清理）。删除/改 tag 前，引擎须算出**全部引用者**（含其它 mod）、**以及"世界已放置方块"**（删方块/物品会让现存存档报错或丢失）并随搁置项展示。

## 5. 已锁定的 v1 决策

- **目标用户**：具备相当魔改知识、但不想手动查写大量脚本的老手；核心痛点是认知负担，靠可视化 UI 解决。
- **输出**：多后端。Intent Spec 是中间表示(IR)，KubeJS / datapack / Almost Unified 各为一个 emitter（编译器多后端模型）。**v1 先实现 KubeJS 后端**（能力最全，可兜底），其余增量补。
- **v1 意图动词**：以 [[04-引擎与动作语义|设计/04 §2]] 的分层为准。引擎原语 `scale / replace_ingredient / remove(hide) / retag / rename`；复合 `unify / harmonize / differentiate / 输入集校正`（"统一"= retag+replace+remove+首选项）；`add_bridge_recipe / change_recipe_type / add_item / remove_item` 仅探测+搁置。
- **数据策略**：离线优先 hybrid——复用已有离线 JAR 分析器（`com.delightify.modinspector`）建静态目录；runtime/WebSocket 仅作可选实时校正通道，不进 v1 关键路径。

## 5b. 例子暴露的扩展能力（超出已锁定 v1 动词，分阶段引入）

> 来源：[[02-炸物食品合成链优化]]。它揭示了 01 没覆盖的几点。

**① "优化" 不是一个动作，而是一族目标。** Agent 须先判断本次"优化"求的是哪一种（可多种并存）：

| 目标模式 | 含义 | 体现 |
|---|---|---|
| 语义贴切 | 用语义最佳的中间产物 | 01 主线（小麦→面粉、煮面用生面条） |
| 合理性/常识 | 输入必须合常识 | 不许"余烬面粉/玉米面粉"做炸鸡；脂肪≠油 |
| 完备性 | 所有合理输入都应可用 | 常识内能炸鸡的油都要可用 |
| 一致性 | 同类配方相互对齐 | 用"鸡腿"而非"生鸡肉"，与其它鸡腿料理一致 |
| 风格贴合 | 符合整合包风格 | 写实沉浸风 → 排除不真实的输入 |

**② 新增动作（超出 `scale / replace / remove / retag`）：**

| 新动作 | 说明 | v1 处置建议 |
|---|---|---|
| **输入集校正** | 收窄（剔除不合理的 tag 成员）或扩充（确保合理项都可用）某输入槽的候选集 | **可进 v1**（retag/replace 家族可表达） |
| **跨配方一致性对齐** | 把离群配方对齐到同类家族 | **可进 v1**（经 replace + 改类型 实现，依赖"配方家族"识别） |
| **新增桥接配方** | 插入中间转化步骤（如 脂肪→荤油），让"直接误用"变成合理加工链 | **v1 仅探测+搁置建议**，不自动新建（高爆炸半径） |
| **改配方类型/处理方式** | 把配方移到更合适的机器/类型（如移出"厨锅"） | **v1 仅探测+搁置建议** |

**③ 配方等价类（跨 mod/机制）：** 同一逻辑步骤常以"沉浸手作"(搅拌碗) 与"自动批量"(搅拌机) 两种形态出现在不同 mod。Agent 应识别为同一等价类、一致处理——这是 §5 "同类物品"在**配方层**的对应物。

**④ 风格档案 = 有效性闸门：** 不再只是偏好，而是驱动合理性校验的约束（写实风 ⇒ 余烬面粉无效、脂肪非油）。要求风格档案能编码"该风格下什么算合理输入"。

**⑤ 聚合级检查：** 有些判断是全局而非逐案的（如"防止'厨锅'类型被泛滥使用"），需在决策清单之上再做一次跨配方的聚合审视。

**⑥ 跨产品模式传播 + 新增物品：** Agent 可从同品类已有成员归纳"规范链模板"（如炸鸡腿的 生→裹糊→油炸），检测偏离的兄弟产品（如只有一步的炸虾），并**建议补齐**——这可能需要**新增当前不存在的中间物品**（如"裹糊虾"）。`add_item` 是迄今**最高爆炸半径**的动作（创建全新注册内容），**v1 仅探测+建议**；落地经 IDE「数据驱动物品模板」生成 + emitter 注册，是 **Agent → IDE 创作工具 → 输出层** 的衔接点。来源：[[02-炸物食品合成链优化|例02]] 口头补充。

**⑦ 整合 ≠ 抹平（来自 [[03-统一所有铜锭|例03]] / [[05-清理重复装饰石|例05]] 批注）：** 面对"统一/清理/优化一批相似物"，**collapse 成一个往往是错的**。Agent 须先判断"真冗余"还是"有意义的区分"，再在三种策略中选：
> - **统一 (unify)**：语义真等价 → 合并（如 4 种纯铜锭）。
> - **区分 (differentiate)**：变体其实是有意义的梯度/品类 → **保留并使区分可见**——如据配方链上下文识别"低纯铜/高纯铜"；如把大量**同名**装饰石**改名为不同种类变体**供玩家区分，同时**一体化其配方链**。
> - **保留惯用差异**：别抹平有意义的机制差异（如 02 的"水桶 vs 流体水"）。
>
> 引出新动作 `rename/relabel`（改显示名消歧）与 `differentiate` 策略。

**⑧ 引导式规划模式（来自 [[04-降低前期配方成本|例04]] 批注）：** 当请求**过于宽泛、几乎全是设计判断(c)**（如"前期太肝，降一档"），Agent **不允许自行做出全套规划并执行**——而是**整合包内相关信息（成本/依赖深度/候选项…）并呈现，引导作者自己规划**。此时 Agent = **决策支持 / 信息整合器**，不是规划者。这是 §3 管线最前面"分流"的那一支，也是 (c) 占比过高时的正确姿态。

## 6. 安全纪律（不可妥协）

(a)(b) 的正确性**不来自"AI 很聪明"，而来自这套纪律**——任一环偷懒，就退化成评审最怕的"不可控系统"：

1. 分类前置（判断可审计）
2. 置信 × 风险双门
3. 不确定即搁置，**绝不静默猜测**
4. 决策清单全透明（连高置信项也逐条可推翻）
5. dry-run + 完整 diff，应用前必经闸门2
6. 可逆 / 可回滚

## 7. 待打磨（靠 [[_如何写例子|例子]] 定）

- 分类法（配方/物品要分成哪些语义类别？粒度多细？）
- 多信号置信的具体权重与冲突判定
- 搁置的触发条件（什么程度的不确定才搁置）
- 风格档案的形态（作者一次性声明 vs Agent 提议确认）
- 置信阈值与风险分级的默认值（保守包/实验包可调）
- 阶段0 目标解释：自动推进 vs 先对齐
- **「优化」目标族**：如何让 Agent 推断本次求的是哪种目标模式（语义贴切/合理性/完备性/一致性/风格）？
- **输入集校正怎么建模**：编辑 tag 成员 vs 编辑配方 ingredient 列表？
- **新增桥接配方 / 改配方类型**：何时从"探测+搁置"升级为可执行动作？
- **配方等价类的识别信号**：怎么判定两条配方是"同一逻辑步骤"（产物相同 + 机制不同）？
- **风格档案如何编码"合理输入"约束**（让"写实风⇒余烬面粉无效"可计算）？
- **聚合级检查放在管线哪一步**（防滥用某配方类型）？
