# MVP 与实施路线图 —— v0 草案

> **状态**：v0 草案。把规格转成**分阶段、可验收**的实施路线（**本规划不写代码**；这是日后开工的地图）。每阶段以前一阶段的「完成定义(DoD)」为门禁；**安全纪律从第一天起不可妥协**。

---

## 1. 总思路

**走通脊柱，再长肉**。最小闭环先端到端打通"摄取 → 图谱 → 一个工作流 → 可审 diff → 产物"，且**完全不依赖未建的高风险件**（runtime / 向量 / 嵌入）。之后每层叠加都先过门禁。

---

## 2. MVP-0：验证产品脊柱

**范围**：
- 离线灌入**一个真实整合包** → 目录+图谱（物品/方块用现有分析器；**配方/tag/lang 用新增 JSON 提取**，见 [[06-数据层]]）。
- 带**缓存材质**的目录/配方浏览（[[08-UI与交互]]）。
- 端到端跑通**一个工作流：统一同名物品**（unify）。选它因为：tag/图逻辑为主、非 AI 重度、Almost Unified 已证明需求并提供已知良好的输出对照。
- 产出**可审 diff** → 作者确认 → **KubeJS 产物**（[[07-输出层-KubeJS]]）。
- **不碰** runtime / 向量 / 嵌入 / add_item。

**完成定义(DoD)**：对一个真实包，能从"统一所有铜锭"这句话，经决策清单审阅（含形近异物搁置）+ diff，生成可 `/reload` 生效的 KubeJS，且引用全部正确、可一键撤销。对照 [[03-统一所有铜锭|例子03]] 跑通。

## 5. 明确推迟 / 不做（及重启条件）

- **runtime Exporter mod + WebSocket**：推迟；重启条件 = 离线缺口（KubeJS 动态/运行时合并）成为真实瓶颈（M6）。
- **向量检索 / ONNX 本地嵌入**：推迟；重启条件 = 词法检索不够（M7）。
- **datapack / Almost Unified 后端**：v1.x；v1 用 KubeJS 兜底。
- **add_item 实际生成（物品模板）**：单独立项；v1 仅探测+建议。
- 这些都**不在 v1 关键路径**，避免拖累脊柱。

## 6. "可直接开工"的判定（本规划的终点）

满足即认为规格已可交付实现：
1. `设计/03–10` 完整、且与 `设计/01-02` 不矛盾（一致性巡检通过）。
2. `例子/01-05` 每个都能在规格里全程走通（行为可复现）。
3. MVP-0 范围、DoD、复用/新建清单无歧义。
4. 5 个核心决策已锁定（✅ 已全部锁定）。
