# 动作词汇表与分类法（v0 草案）

> **状态**：v0 草案，从 [[01-优化所有面食配方|例子01]] 与 [[02-炸物食品合成链优化|例子02]] 提炼。活文档，随例子增补。
> 这是通往「Intent Spec / 决策清单规范」的**前置物**——先把"Agent 能做哪些动作、怎么分类、怎么圈定对象"盘清楚，再去定那份契约的字段。

---

## 1. 三个层次（先理清概念，别混）

| 层次 | 是什么 | 归谁 |
|---|---|---|
| **目标模式 Goal Mode** | 这次任务求的是什么效果（求一致？求合理？） | 作者给/确认（属 (c)） |
| **分类 Classification** | Agent 给每个对象贴的语义标签（"这是煮面步骤"） | Agent 产出，人可审 |
| **动作 Action** | 落到数据上的机械操作（替换/缩放/…） | 引擎确定性执行 |

管线关系：**目标模式**框定要达成什么 → 逐案**分类**判断这是什么 → 据分类+目标选**动作**。

---

## 2. 目标模式清单（求什么）

| 目标模式 | 含义 | 来源 |
|---|---|---|
| 语义贴切 | 用语义最佳的中间产物 | 例01 |
| 合理性/常识 | 输入合常识（脂肪≠油） | 例02 |
| 完备性 | 所有合理项都应可用 | 例02 |
| 一致性 | 同类配方相互对齐 | 例02 |
| 风格贴合 | 符合整合包风格档案 | 例01/02 |
| 去重/统一 | 同类物品归一 | 例01 |
| 数值平衡 | 成本/时间/产出缩放 | v1 动词 |
| 清理 | 移除/隐藏冗余 | v1 动词 |

一次任务可命中多个目标模式。**待打磨：Agent 如何从"优化所有面食"这种话推断目标模式组合？**（见 [[01-核心设计与决策模型|§7]]）

## 3. 动作词汇表（怎么改）

> 选择器 = 动作作用的对象集合（见 §5）。风险轴决定置信×风险门里的"风险"。

**引擎原语（low-level）：**

| 动作 | 语义 | 风险/可逆 | v1 | 来源 |
|---|---|---|---|---|
| `replace_ingredient` 替换原料 | 配方内 A→B（按物品或 tag） | 低 / 可逆 | ✅ | 例01/02 |
| `scale` 数值缩放 | 输入量/时间/产出 ×N 或 ±% | 低 / 可逆 | ✅ | v1 |
| `retag` 改 tag 归属 | 物品并入/移出 tag | 中高 / 跨配方影响 | ✅ | v1 |
| `remove`/`hide` 移除/隐藏 | 去掉配方或物品 | 高 / 难逆 | ✅（默认搁置） | v1/例01 |
| `constrain_inputs` 输入集校正 | 收窄（剔除不合理）或扩充（补齐合理）某输入槽候选集 | 中 / 视方向 | ✅ | 例02 |
| `rename`/`relabel` 改显示名 | 改物品显示名以消歧（如把多个"大理石"改成不同种类变体名） | 低 / 可逆（仅 lang） | ✅ | 例05批注 |

**复合/编排（high-level，由原语组成）：**

| 动作 | 由什么组成 | v1 | 来源 |
|---|---|---|---|
| `unify` 统一同类 | retag + replace + remove + 首选项 | ✅ | 例01 |
| `harmonize` 一致性对齐 | replace + (change_type) 对齐到同类家族 | ✅（不含改类型部分） | 例02 |
| `differentiate` 区分 | 识别"伪同类"实为梯度/变体 → retag + rename + 配方链整合，使区分可见而非合并 | ✅ | 例03/05批注 |

> **整合 ≠ 抹平**：遇"统一/清理"先判断该 `unify`（真等价）还是 `differentiate`（有意义的区分），别一律 collapse。详见 [[01-核心设计与决策模型|§5b⑦]]。

**探测+搁置（v1 只发现并建议，不自动执行）：**

| 动作 | 为何先不自动 | 来源 |
|---|---|---|
| `add_bridge_recipe` 新增桥接配方 | 凭空造配方爆炸半径高、易错 | 例02（脂肪→荤油） |
| `change_recipe_type` 改配方类型 | 换机器/类型涉及该 mod 机制假设 | 例02（移出厨锅） |
| `add_item` 新增物品 | 创建当前不存在的中间物品（如"裹糊虾"）补齐规范链——全新注册内容、爆炸半径最高 | 例02补充 |

> `add_item` 经 IDE「数据驱动物品模板」生成（Agent 建议 id/名/tag/所属链，IDE 脚手架，emitter 注册）；新物品材质/模型可模板派生或需作者补图。这是 Agent → IDE 创作工具 → 输出层的衔接点。

**聚合级（分析，非逐案）：**

| 检查 | 作用 | 来源 |
|---|---|---|
| `aggregate_audit` 聚合审视 | 全局模式检查，如"某配方类型被滥用" | 例02 |

## 4. 分类法（怎么判断"这是什么"）—— 草案，待更多例子

Agent 在动作前给对象贴标签。目前从两例可归纳几个维度（粒度待定）：

- **语义域归属**：目标域内（该处理）vs 域外（跳过）。例：魔法转换 = 烹饪域外。
- **链中角色**：原料 / 中间产物步骤 / 成品步骤。例：制面团 vs 煮面 vs 蒸馒头。
- **机制形态**：沉浸手作 / 自动批量 / 魔法 / 原版 → 用于识别**配方等价类**（同角色不同形态 = 同一逻辑步骤）。
- **优化状态**：已合理 / 待优化（输入不当、类型不当、不一致）。
- **真冗余 vs 有意义区分（伪同类）**：一组相似物是"该合并的真冗余"还是"该保留的梯度/变体"——决定 `unify` vs `differentiate`（例03/05批注）。
- **形近异物的处置是 context-dependent**（例03批注）：默认不并入，但**具体配方的产物若语义上能接受相近物，则交用户抉择（搁置）**，而非一律跳过。

> **待打磨**：分类法该多细？固定枚举还是开放标签？这是后续例子要回答的核心问题。

## 5. 选择器需求（怎么圈定对象）

召回与逐案都要能按这些维度寻址对象集：

- 按名称/显示名语义（lang）、按 tag、按 mod、按配方类型/机器
- 按图谱位置（上游/下游、引用者）
- 按**产物语义家族**（"所有鸡腿料理"）—— 例02 一致性对齐需要
- 按**配方等价类**（同一逻辑步骤的不同形态）—— 例02 需要
- 按**品类产品族 + 规范链模板**（"所有炸制食品"该共享 生→裹糊→油炸）—— 跨产品模式传播需要（例02补充）

后两者比 [[01-核心设计与决策模型]] 设想的选择器更"语义"，可能需 LLM 辅助成组，而非纯规则。

## 6. 待作者拍板

1. **v1 动作范围**：认可"输入集校正 + 一致性对齐进 v1，新增桥接配方 + 改配方类型仅探测+搁置"吗？
2. **分类法形态**：倾向**固定枚举**（可控、可测）还是**开放标签**（灵活、难测）？我建议 v1 用一套小枚举 + 允许 Agent 标注"其它(说明)"。
3. **目标模式**：要不要让作者在任务时**显式可选**目标模式（勾"求一致性/求合理性"）？我倾向"Agent 推断 + 作为头号可改假设展示"，并提供显式勾选作捷径。

## 7. 建议的下一批例子

- **纯 v1 动词类**：一个"统一所有铜锭"（unify）、一个"所有 Create 配方成本-20%"（scale）——把 §3 的 v1 原语各自跑通。
- **一致性对齐专项**：一个明确以"让同类配方风格一致"为目标的例子，校准 harmonize + 配方家族识别。
- **目标模式冲突**：一个"合理性"与"完备性"打架的例子（既要排除不合理的油、又要保证合理的油都在）——看 Agent 怎么权衡/搁置。
