# 例子 05：同名装饰石的消歧与链整合（rename + differentiate）（据作者批注重写）

> **据作者批注修订**。原拟稿把需求设成"去重装饰石、只留一套"——作者指出这**不合逻辑**：装饰只要长得不一样就有保留价值。真需求是：大量**同名**装饰石**改名为不同种类变体 + 一体化配方链**（消歧，而非删除）。原拟稿 + 批注见文末附录。

## 0. 我理解的你的意图（请核对）

好几个 mod 都加了显示名都叫"大理石"的方块——**真问题不是重复，是同名导致分不清**。所以 Agent 应当：
1. **改名消歧（rename）**：据各自纹理/来源把它们改成**不同的种类变体名**，让玩家一眼区分；
2. **配方链一体化（differentiate + 链整合）**：把它们的获取/加工链规整成一致、完整（缺配方的补齐，必要时可互转）。

**保留多样性、让它可辨识**——绝不是删除、也不是合并成一个。这是 [[01-核心设计与决策模型|§5b⑦]]「整合≠抹平」的标准范例。

## 1. 用户目标（输入）

> "整合包里好几个 mod 都加了叫'大理石'的装饰方块，名字一样根本分不清，帮我理一下"

## 2. 整合包上下文（id 仅示意）

### 相关物品
| id（显示名） | 语义 |
|---|---|
| `create:marble`（大理石） | 纹理偏云白；**有功能用途**（参与 Create 雕饰/配方） |
| `quark:marble`（大理石） | 纹理偏灰白；纯装饰 |
| `somemod:marble`（大理石） | 又一种纹理；纯装饰，**无获取配方**（仅创造可得） |
| 各套的台阶/楼梯/墙 | 衍生件，显示名同样撞车 |

## 3. 期望的 Agent 行为（逐案）

| # | 对象 | 分类 | 决策 | 理由 | 置信·风险 | 处置 |
|---|---|---|---|---|---|---|
| 0 | 命名风格 | 作者判断 | 默认按"视觉特征/来源"改名（如"云纹大理石/灰岩大理石…"），可改 | 命名属 (c) | —·— | ⚠️ 搁置（头号） |
| 1 | N 个同名"大理石" | 同名伪冲突（**非真冗余**） | **rename** 成不同种类变体名 | 视觉不同即有价值，问题在分不清 | 中·低（仅 lang） | ✅ 自动（名字可审） |
| 2 | 各自获取链参差（有的可切石、有的无配方） | 链不完整/不一致 | **一体化**：补齐缺失获取、规整为一致加工链 | 即"配方链一体化" | 中·中 | ⚠️ 搁置 |
| 3 | 衍生件（台阶/楼梯/墙） | 跟随 | 改名跟随主方块、链一致 | 保持一套内部一致 | 中·低 | ✅ 自动（随主方块） |
| 4 | 🪤 `create:marble`（有功能用途） | 有功能 | 改名安全（动 lang 不动 id），但链整合时**保留其功能引用** | 改显示名不影响 id 引用 | 高·低 | ✅ 自动（注意保功能） |

## 4. 期望进入"待裁决队列"的项

- **命名风格 / 具体新名**（默认给一套，可改）。
- **配方链一体化到什么程度**：仅补齐缺失获取 / 规整为一致加工 / 强制全部可互转。

## 5. 边界 / 不该做的事（反例）

- **不要**删除任何一种"大理石"——视觉不同即有价值。
- **不要**把它们合并成一个——这是 `differentiate`（区分），不是 `unify`（合并）。
- **rename 只改显示名（lang），不动注册 id**——动 id 会破坏存档与跨 mod 引用。

## 6. 备注 / 开放问题

- `rename` 只改 lang → **低风险、可逆**，是 v1 安全动作。
- 标准的"整合≠抹平"：**链上 unify、视觉上 preserve + clarify**。
- 待打磨：判断"该 differentiate 而非 unify"的信号——同显示名 + 纹理/模型不同 + 各有独立世界生成，是强信号。

**原拟稿前提（已被否）**：目标设为"清理重复装饰石、只留一套"，用 `remove/hide` 去重（默认隐藏、保留 create:marble、移除需警告"世界已放置方块"）。

**作者批注（verbatim）**：
> 这种需求实际上不合逻辑，"装饰"只要不是长得完全一样就没有只留一个的逻辑，更常见的需求是针对大量同名装饰石，由Agent进行配方链条的一体化，并将资源名称更改为不同的种类变体供玩家区分
