# 例子 04：降低前期配方成本 —— 引导式规划范例（据作者批注重写）

> **据作者批注修订**。原拟稿把"前期太肝"当成可自动规划执行的任务——这是**错的**。作者指出：这种过宽、几乎全是设计判断 (c) 的需求，Agent **不允许自行规划执行**，应转**引导式规划**。原拟稿 + 批注见文末附录。

## 0. 我理解的你的意图（请核对）

"Create 前期太肝，降一档"是一个**过宽、几乎全是设计判断 (c)** 的需求——"哪些算前期""肝在哪""降多少""降输入还是加产出"全是你的判断。所以 Agent **不规划、不执行、不替你选**，而是切换到**引导式规划**：把整合包里跟"前期成本"相关的信息整理成一张**只读的决策支持视图**，帮你一眼看清、由你来定。等你在视图上圈定范围+设好降幅，**那一刻**才生成一个明确的 `scale` 任务，走正常执行管线。

> 这是 [[01-核心设计与决策模型|§3 管线]] 最前面"分流"的那一支，也是 §5b⑧「引导式规划」的范例。

## 1. 用户目标（输入）

> "Create 前期太肝了，把前期配方的材料成本降一档"

## 2. 整合包上下文（id 仅示意）

Create 早期相关配方若干（安山合金、各机械件…），各有当前成本（输入量）、依赖深度、产出。

## 3. 期望的 Agent 行为 —— 产出"决策支持视图"，不动数据

> ⚠️ 与执行型例子的根本区别：这里 Agent 的产出是**信息 + 选项（只读）**，不是决策清单，**不产生任何变更**。

**A. "前期"有哪几种界定口径**（因为"前期"本身没定义）：

| 口径 | 覆盖配方数 | 说明 |
|---|---|---|
| 依赖深度 ≤2 | ~30 | 从原料起两步内 |
| "引入电力之前" | ~45 | 机械动力但未通电阶段 |
| 你手动圈定 | — | 最精确，推荐 |

**B. 候选配方的成本/深度 + 疑似"肝点"排序**：

| 配方 | 依赖深度 | 当前成本（输入） | 疑似肝点？ |
|---|---|---|---|
| 安山合金 | 1 | 安山岩 + 铁块/铁锭 | ★★ 高（前期海量需求） |
| …机械件 | 2 | … | ★ |

**C. 可调旋钮（由你设定）**：降幅（×0.9 / ×0.8 / …）· 降输入 or 增产出 · 取整规则。

**D. Agent 明确声明**：不会自己选范围、不会自己定降幅——这些等你定。

→ **交接**：你在视图上①圈定范围②设定降幅后，这部分**才**变成明确的 `scale` 任务，进入正常执行管线（逐案、置信×风险、搁置；且仍守"1:1 守恒转化不缩放"等机械规则）。

## 4. 期望进入"待裁决队列"的项

引导阶段**没有**待裁决项——因为还没产生任何决策。所有判断都在视图上交还给你。（待裁决只在交接后的执行阶段才可能出现，如某配方该不该算成本型。）

## 5. 边界 / 不该做的事（反例）

- **不要**自行选定"前期"范围就改——范围必须你在视图上圈。
- **不要**自行决定降幅 / 降输入还是加产出。
- **引导阶段只读**：呈现信息、不产生任何数据变更。
- 进入执行后仍守机械规则：**不缩放 1:1 守恒转化**。

## 6. 备注 / 开放问题

- 本例确立了**引导式规划的产物形态 = 只读决策支持视图**（信息+选项+旋钮），与执行型例子（决策清单）根本不同。
- 待打磨：**触发条件**——Agent 怎么判定一个请求"过宽到该走引导而非执行"？（候选信号：(c) 占比高、范围/幅度均未指定、影响面巨大。）

## 附录：原拟稿 + 作者批注（保留）

**原拟稿前提（已被否）**：把"前期太肝降一档"当作可执行任务，Agent 给默认范围（深度≤2）+默认降幅（×0.7）作为"头号假设"，然后逐案 `scale` 执行（守恒型 1:1 跳过、后期配方跳过）。

**作者批注（verbatim）**：
> 这种过于宽泛和自由的需求是不符合我们设计思想的，此时不允许Agent自行做出全套规划并进行，而是通过整合整合包内相关信息引导用户进行规划。
