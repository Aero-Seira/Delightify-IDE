# 例子 02：优化油炸食品合成链（炸鸡腿为例）

> **状态**：作者草稿 → Claude 精修（供确认）。我把意图显式化、补全了空缺列；**凡我推断/补充处标【补】**，请重点核对。原始草稿完整保留在文末「附录」。

---

## 0. 我理解的你的意图（请先核对这段）

"炸鸡腿"是一条 **裹糊 → 油炸** 的两步链，且在包里以**手作沉浸版**（搅拌碗→炸锅）和**自动批量版**（搅拌机→机械炸锅）平行存在。你要的"优化"= 三件事：

1. **审计每步输入是否合理**：裹糊步该用正经面粉（别让"余烬面粉/玉米面粉"混进来）；油炸步该用真正的"油"（脂肪不能直接当油，但可补"脂肪→荤油"的桥）。
2. **平行链语义一致、但不抹平机制差异**：两套链该用同一套语义（同一面粉概念、同一鸡腿），但各自的机制惯用法（水桶 vs 流体水、手操 vs 批量）要保留。
3. **纠正退化配方**：那条 `生鸡肉 --厨锅--> 鸡腿` 抄近道、绕过了正规链，要拉回（换输入、必要时换类型），并警惕"厨锅"被当万能锅滥用。

> 前提：风格档案 = **写实沉浸向**——这是上面所有"合理性"判断的依据（换个魔法向包，结论可能不同）。

## 5. 边界 / 不该做的事（反例）【补】

- **不要**抹平 #1 与 #2 的机制差异——水桶 vs 流体水是惯用法，不是错误。
- **不要**把脂肪静默当油，也**不要**静默新增"脂肪→荤油"桥接——桥接须经确认。
- **不要**因 id 文本相近就认定 #5 产物与正规炸鸡腿等同——要看链与语义。

## 6. 备注 / 开放问题

- "余烬面粉/玉米面粉算不算不合理"完全取决于风格档案——本例最依赖你确认的"写实"前提。
- 【补】"荤油"是否真有 mod 提供？若无，桥接建议不成立，退化为"仅排除脂肪"。
- 置信普遍中/低 → 本例多数项默认搁置，符合"宁可多问"。

## 7. 扩展场景【补，2026-06-12 口头补充】：跨产品的模式传播

假设包里还有别的"炸制食品"，如**天妇罗炸虾**，但某 mod 只提供了**直接一步链** `虾 --[炸]--> 炸虾`（没有裹糊步）。在"裹糊鸡腿"那条规范链已存在的前提下，期望 Agent：

- **感知结构差异**：同属"炸制食品"品类，炸鸡腿走 `生→裹糊→油炸` 两步规范链，炸虾却只有一步——这是**品类内的不一致 / 退化链**。
- **以炸鸡腿链为模板，建议把炸虾规范化**成 `虾 → 裹糊虾 → 炸虾`。
- 这需要一个**当前不存在的中间物品「裹糊虾」**——触发新动作 **新增物品（add_item）**。

| 对象 | 分类 | 期望决策 | 理由 | 置信·风险 | 处置 |
|---|---|---|---|---|---|
| `虾 --[炸]--> 炸虾`（一步） | 炸制食品·缺裹糊步 | 建议**新增「裹糊虾」物品 + 补成两步链**，与炸鸡腿模板对齐 | 同品类应共享规范链；缺裹糊步是退化 | 中·**极高** | ⚠️ 搁置（仅建议，绝不自动建物品） |

**落地方式（你指出的）**：新增物品经 IDE 的**「数据驱动物品模板」**生成——Agent 给出 id / 显示名 / tag / 所属链的建议，IDE 用模板脚手架创建，emitter(KubeJS) 产出注册代码。这是 **Agent → IDE 创作工具 → 输出层** 的衔接点。

> 诚实提示：新物品需要**材质/模型**——模板可自动派生（如在虾贴图上叠面糊色）或需作者补图。这是"新增物品"最现实的成本，也是它必须搁置确认的又一理由。
