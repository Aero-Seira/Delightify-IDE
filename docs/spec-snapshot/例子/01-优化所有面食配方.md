# 例子 01：优化整合包内所有面食的配方

> 这是与 Claude 共同推导出的**范例**，照此格式书写即可。它贯穿了"先分类→再动作→置信×风险→搁置"的完整决策模型。

## 1. 用户目标（输入）

> "优化整合包内所有面食的配方"

风格档案（一次性声明，全局复用）：**写实烹饪向整合包，偏好 farm-to-table 的中间产物链条。**

## 2. 整合包上下文

### 相关物品
| id（显示名） | 语义 / 说明 |
|---|---|
| `minecraft:wheat`（小麦） | 原始谷物，未加工 |
| `create:wheat_flour`（面粉） | 小麦研磨产物，制作面食的基础 |
| `create:dough`（面团） | 面粉+水，普通面团 |
| `bakery:fermented_dough`（发酵面团） | 发酵后的面团，更适合做馒头/包子 |
| `farmersdelight:raw_pasta`（生面条） | 通用生面条，上游：面团→擀制 |
| `cuisine:ramen_noodles`（拉面生面） | 拉面专用，上游：含碱水的特殊配方 |
| `cuisine:udon_raw`（乌冬生面） | 乌冬专用，上游：特殊配方 |
| `bakery:mantou`（馒头） | 蒸制面食 |
| `magicmod:bread_of_creation`（造物面包） | 魔法物品 |

### 相关配方
| 配方（输入 --[方式]--> 输出） | 来源 mod | 备注 |
|---|---|---|
| `小麦 --[合成]--> 面团` | somemod | 直接用整粒小麦做面团 |
| `面粉 --[锅煮]--> 熟面条` | cuisine | "煮面"步骤，当前直接用面粉 |
| `面团 --[蒸]--> 馒头` | bakery | |
| `面团 --[魔法转化阵]--> 造物面包` | magicmod | 配方类型 = `magicmod:transmutation` |

### 相关 tag
- `#forge:dough`：包含 `create:dough`、`bakery:fermented_dough`
- `#forge:flour`：包含 `create:wheat_flour`

## 3. 期望的 Agent 行为（逐案）

> 阶段0 假设：把"优化"理解为「让面食链使用语义最贴切的中间产物，并合并功能重复的物品」。
> 阶段A 范围：经 `面食` 展开（flour/dough/noodle/馒头/pasta…）命中 4 条候选配方、9 个相关物品。

| #   | 对象                     | 期望分类      | 期望决策                                 | 理由（语义线索）                                      | 置信·风险   | 处置       |
| --- | ---------------------- | --------- | ------------------------------------ | --------------------------------------------- | ------- | -------- |
| 1   | `小麦 --[合成]--> 面团`      | 制作面团步骤    | 输入 小麦 → 面粉                           | 面团语义上应源于面粉而非整粒小麦；存在 `create:wheat_flour`      | 高·低     | ✅ 自动     |
| 2   | `面粉 --[锅煮]--> 熟面条`     | 煮面步骤      | 输入 面粉 → **生面条**                      | "煮面"的输入应是生面条，不是面粉；这是最易被一刀切规则改错的陷阱             | 高·低     | ✅ 自动     |
| 3   | ↳ 但有 3 种生面条            | 选型        | 暂选 `farmersdelight:raw_pasta`（通用生面条） | 拉面/乌冬有各自专门上游配方与名称语义，通用煮制配方应配通用生面条             | 中·低     | ⚠️ 搁置    |
| 4   | `面团 --[蒸]--> 馒头`       | 蒸制面食      | 面团 → **发酵面团**                        | 馒头是发酵面食，`bakery:fermented_dough` 语义更贴合        | 中高·低    | ✅ 自动（可审） |
| 5   | ↳ #4 的连带后果             | 冗余清理      | 提议移除 `create:dough` 在本链中的冗余引用 / 物品   | 替换为发酵面团后，普通面团在该链无引用，且与 `#forge:dough` 重叠      | —·**高** | ⚠️ 搁置    |
| 6   | `面团 --[魔法转化阵]--> 造物面包` | 魔法转换（非烹饪） | **不替换**                              | 配方类型 `magicmod:transmutation` 表明这是魔法机制，非常规烹饪链 | 高·—     | ⏭️ 跳过    |

## 4. 期望进入"待裁决队列"的项

- **#3 用哪种生面条？** 选项：`farmersdelight:raw_pasta`（通用）/ `cuisine:ramen_noodles`（拉面）/ `cuisine:udon_raw`（乌冬）。暂定推荐 = 通用生面条。为何不确定 = 多个语义相近物品，"煮面"未指定面种。
- **#5 是否移除冗余面团物品？** 选项：移除 / 保留但去 tag / 不动。暂定推荐 = 保留物品但从本链去引用（移除物品爆炸半径高）。为何不确定 = `create:dough` 可能被其它 mod 配方引用，删除需看完整引用清单。
- **阶段0「优化」的定义**：若你指的是降成本 / 减步骤 / 平衡难度而非"语义贴切"，请改。

## 5. 边界 / 不该做的事（反例）

- **不要**因为"统一面食材料"就把 #6 的魔法配方也改了——魔法转换不属于烹饪语义链。
- **不要**用一条全局规则（如"所有面食配方的小麦/面粉都换成 X"）一刀切——#1 该用面粉、#2 该用生面条，语义不同。
- **不要**静默删除 `create:dough`——任何删除都要先列出全部引用者并搁置确认。

## 6. 备注 / 开放问题

- "馒头该用发酵面团"算高置信还是中置信？取决于我们对"发酵面食"语义识别的把握——这是个值得多写几个例子去校准的点。
- 风格档案若改成"魔法向整合包"，#6 是否反而应当参与某种统一？（风格如何影响决策，需要专门的例子。）
